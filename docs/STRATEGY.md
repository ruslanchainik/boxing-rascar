## 1. Что реально считается в метрике

Формула на одно видео:

```
final = 0.50*time + 0.20*fighter + 0.10*punch_type + 0.08*effectiveness
      + 0.06*hand + 0.06*target − fp_penalty
fp_penalty = N_fp / (N_gt + N_fp)
```

Матчинг — венгерский алгоритм по `|Δt| ≤ 1.0 с`, `t = frame/30`. `time_error = min(1, |Δt|/0.5)` — то есть **ошибка <0.5 с даёт линейный буст по 50% score**, а >0.5 с обнуляет атрибуты.

### Что это значит на практике

| Компонент | Доля | Что максимизирует |
|---|---:|---|
| `time` | 50% | Точность пика кадра удара (~±15 кадров критично) |
| `fighter` | 20% | Устойчивая red/blue идентификация на всём раунде |
| `punch_type` | 10% | Balanced acc по {jab, cross, hook, uppercut} с весами |
| `effectiveness` | 8% | Balanced acc по {hit, block, miss} с весами |
| `hand` | 6% | left/right |
| `target` | 6% | head/body |
| FP-penalty | — | Лучше пропустить, чем нагенерить |

**Главные выводы:**

1. Детектор момента удара = главное. 50% метрики и зависимость атрибутов от `time_error<1`. Любая модель должна предсказывать *peak frame* с разрешением 5–15 кадров.
2. `fighter` (20%) — это task о треке, не о цвете. Цвет — только инициализация.
3. **Precision важнее recall.** FP-штраф квадратично растёт. Если выбираем порог, целимся в precision ≥ 0.75 даже ценой recall ≈ 0.6 — это часто выгоднее.
4. `clear=false` строки бесплатны. Можно набивать sample_submission ими, ничего не теряя.
5. Атрибуты `punch_type/effectiveness` взвешены обратной частотой → угадывание редких классов (`uppercut`, `blocked`) даёт непропорциональный буст.

## 2. Особенности данных

- **63 train видео, 9030 ударов, 9 test (3 боя × 3 раунда).** Очень мало для end-to-end видеомоделей с нуля — обязателен трансфер.
- Test и train — пересекающаяся студия камер (один турнир), но test содержит *боёв-доменов* нет в train (Бой9, Бой2, Бой8). Возможен сильный перекос: красные шорты у одного бойца, синий ринг и т.д. Регуляризация и color-augment обязательны.
- Тренировочные видео (`бокс/IMG_*.mp4`) имеют нейтральный цвет экипировки. Тестовая выборка вся турнирная → можно дообучаться только на турнирной части, а тренировочную использовать как general action prior.
- `fps` в CSV пуст. Метрика жёстко использует 30 fps. Реальный fps может быть 25/29.97/50/60 — **обязательно перевести предсказанные кадры в "frame at 30 fps" семантике метрики**. То есть `t = real_frame / real_fps`, потом `metric_frame = round(t * 30)`. Это легко даёт +X% если сейчас игнорируется.
- Длительность раундов в боксе ~2 мин. На 30 fps это 3600 кадров. 9030 ударов / 63 видео ≈ 143 удара/раунд — *очень плотно*. Среднее расстояние между ударами ~25 кадров (~0.83 с). Окно матча ±1 с → соседние удары конкурируют за матчинг. Это значит **NMS/peak-picking на временной оси критичен и должен быть аккуратнее, чем обычное `min_distance=30`**.

## 3. SOTA-подходы по семействам

### 3.1 Action spotting (главное семейство для этой задачи)

Эта задача формально — *action spotting*: дано длинное видео, предсказать (timestamp, class). Литература:

- **NetVLAD++ / Pooling** (SoccerNet 2018-2020). Slide-window CNN-фичи + temporal pooling. Лёгкий, простой baseline.
- **Temporally-Aware Feature Pooling (TAP)**, **CALF context-aware loss** — учат "пик в центре окна".
- **E2E-Spot / Spot** (Hong et al. 2022) — end-to-end frame-level classifier с GSM/GSF temporal blocks; SOTA на SoccerNet-v2/Tennis. **Очень подходит**: regnety_002 + GRU, обучается быстро, выдаёт per-frame logits → peak picking.
- **T-DEED** (CVPR 2024) — улучшение E2E-Spot с deformable encoder. Сейчас SOTA на FigureSkating/FineGym/Tennis. Высокая точность по времени.
- **ASFormer / ActionFormer** — трансформер по temporal feature pyramid, отлично подходит для дальних зависимостей. Изначально для action detection (interval), но легко конвертится в spotting.
- **PointTAD / TriDet** — anchor-free детекция действий по времени.

**Что выбрать первым:** E2E-Spot (или T-DEED), потому что (а) есть код, (б) end-to-end, (в) понятная peak-picking логика.

### 3.2 Video classification backbones для фичей

- **VideoMAE v2** (CVPR 2023). Self-supervised pretrain на Kinetics-700+UnlabelledHybrid. Лучшее качество на трансфер для маленьких датасетов.
- **InternVideo / InternVideo2** (2023-2024). Multi-modal, очень сильный transfer.
- **UniFormerV2**, **MViTv2** — компактнее, дешевле.
- **Video Swin** — стабильная классика.
- **X3D-M/L** — лёгкий, реал-тайм, для слабого железа.

Сценарий: гонишь окна 16–32 кадров через backbone → берёшь токены → подаёшь во temporal head (Transformer/Conv1D).

### 3.3 Pose / skeleton

- **YOLO-pose / RTMPose** — детекция keypoints (уже есть в baseline).
- **MMPose RTMW / DWPose** — точнее, особенно по кистям. Для удара критичны wrists.
- **ST-GCN++, MS-G3D, CTR-GCN** — skeleton-based action models. На 1-секундных клипах работают очень быстро.
- **PoseConv3D** (Duan, ICCV 2021) — рисует heatmaps keypoints → 3D CNN. SOTA на NTU/Volleyball, отлично на малых данных.
- **PoseFormerV2** — трансформер по keypoints.

Pose-фичи особенно полезны для `hand`, `punch_type`, `target` (left/right + траектория кисти → классы). И как сигнал для `time` (peak velocity wrist).

### 3.4 Optical flow / motion

- **RAFT / GMA** — flow в роли второго стрима. Очень хорошо для детекции момента удара (резкий пик кисти).
- Дешёвый вариант: **dense flow от OpenCV (Farneback) или TVL1** в окне ±0.3 с от потенциального события.
- На малых данных two-stream (RGB + flow) часто бьёт single-stream.

### 3.5 Hand/glove-centric

- Отдельный детектор перчаток (YOLOv8/RT-DETR fine-tune на ~500 ручных боксах) → трек кисти → событие = пик скорости + контакт. Это очень точный сигнал для `time` и `hand` и относительно дешёвый.

### 3.6 Multi-modal & foundation

- **CLIP/SigLIP** zero/few-shot фичи на кадрах для `effectiveness` (landed vs miss) — может зацепить визуальные различия (reaction соперника, отскок).
- **VideoLLaMA / Qwen2.5-VL / Gemini-2** для лейблинга трудных случаев (semi-supervised, pseudo-labels).

## 4. Гипотезы — ранжированные по ROI

Каждая гипотеза: **что**, **зачем**, **сколько часов**, **ожидаемый прирост**.

### H1 — Pose+Skeleton-CNN детектор пика удара (peak frame regression) ★★★★★

**Что.** На каждом кадре строим вектор pose-фич для red и blue (нормализованные координаты wrist/elbow/shoulder/hip, скорости, расстояние wrist→opponent head/body). Подаём окно 64–128 кадров в 1D-CNN/TCN с двумя выходами:
1. heatmap-like per-frame probability события (sigmoid),
2. одновременно — multi-task heads (fighter, type, hand, target, effectiveness) на пике.

Loss: focal на per-frame + Gaussian-target around GT frame (σ=3–5 кадров). Peak-picking: NMS с `min_distance=10`.

**Зачем.** Pose уже даёт baseline; добавление **soft Gaussian-targets вместо hard 0/1** обычно даёт +5-10% time-score. Multi-task head шарит представление и удешевляет атрибуты.

**Стоимость.** Локально на CPU/8 ГБ — реалистично. Skeleton 1D-сеть весит <10 МБ.

**Ожидаемый эффект.** Самый большой ROI на старте. Должно поднять с baseline ~0.2-0.3 до ~0.45-0.55.

### H2 — End-to-end E2E-Spot / T-DEED на RGB ★★★★★

**Что.** Запустить готовый E2E-Spot с regnety_008 backbone + GRU temporal. Вход — 224×224 кропы вокруг ринга, 224×398 или 398×224, fps=12.5. Multi-class head: `bg + jab × 2 hands × 2 fighters + cross × ... = 1 + 4*2*2 = 17 классов` (или плоское multi-task на 4 головы).

**Зачем.** End-to-end, не зависит от качества pose. Хорошо ловит общий контекст (clinch, движение корпусом). Достигает SOTA в spotting на схожих доменах (теннис, фигурное катание).

**Стоимость.** Один A100 на 12-24ч, или 3090 на 1-2 дня. Локально не тренируется (нужно ≥16 ГБ VRAM).

**Ожидаемый эффект.** Сильный ансамбль-партнёр для H1. +0.05-0.10 поверх pose-only.

### H3 — Two-stream: pose + RGB-window classifier на отобранных кандидатах ★★★★

**Что.** Каскад:
1. H1 даёт кандидатов (порог recall 0.9, precision 0.4).
2. На каждом кандидате берём окно ±20 кадров, прогоняем через VideoMAE-v2-S (16 кадров, 224×224, кроп вокруг ринга).
3. VideoMAE отдаёт: is-true-punch, type, effectiveness, hand, target.

**Зачем.** Разделяем "поиск пика" (дешёвая модель на потоке) и "тяжёлую классификацию" (дорогая модель на 0.1% кадров). VideoMAE pretrain даёт качество без больших данных.

**Стоимость.** Inference вполне реальный, train — A100 8-16ч.

**Ожидаемый эффект.** Сильно поднимает атрибуты (`type` +0.05-0.1, `effectiveness` +0.05).

### H4 — Glove tracking + velocity peak ★★★★

**Что.** Отдельный YOLOv8/RT-DETR на перчатки (fine-tune на ~300-500 кадрах с авторазметкой через CLIP-prompt или ручной). Трек перчатки → скорость → пики скорости. Удар = пик скорости + сближение с opponent head/body bbox.

**Зачем.** Прямой физический сигнал. Очень точный по времени (±2-5 кадров). Решает `hand` напрямую.

**Стоимость.** Полдня на разметку + 2-4ч обучения детектора.

**Ожидаемый эффект.** +0.03-0.07 к `time`, почти бесплатное `hand`.

### H5 — Устойчивый red/blue assignment через color+pose+tracking ★★★★

**Что.** Baseline уже разделяет red/blue, но он шумит. Делаем:
1. ByteTrack/BoT-SORT по двум основным трекам.
2. На каждом треке агрегируем patches шорт/майки/перчаток (через keypoints hip, chest, wrist).
3. Считаем HSV-гистограммы → стабильный per-track цвет за всё видео.
4. Глобальное решение: Hungarian assignment треков к {red, blue} по одной оптимизации на видео.

**Зачем.** `fighter` = 20% метрики. Любые мелкие свапы стоят дорого. Глобальная стабильность лучше per-frame решений.

**Стоимость.** 1-2 дня.

**Ожидаемый эффект.** +0.05-0.10 к `fighter` и косвенно к остальному.

### H6 — Frame-rate aware time mapping ★★★★★ (дёшево + критично)

**Что.** Проверить реальный fps каждого test-видео. Сабмитить `frame_at_30fps = round(real_frame / real_fps * 30)`.

**Зачем.** Метрика жёстко использует 30. Если видео 60 fps, ошибка во времени удваивается = атрибуты обнуляются.

**Стоимость.** 30 минут.

**Ожидаемый эффект.** Если сейчас не делается — может быть +0.05-0.15. Обязательно проверить.

### H7 — Class-balanced loss + auxiliary heads ★★★

**Что.** Loss с весами обратной частоты по `punch_type` и `effectiveness` (как в метрике). Plus focal-loss на бинарном детекторе события (positive class редкий: 1 кадр из ~150).

**Зачем.** Метрика прямо использует balanced acc — обучать в той же геометрии.

**Стоимость.** 1 час.

**Ожидаемый эффект.** +0.02-0.05.

### H8 — Optical flow second stream для type/effectiveness ★★★

**Что.** На отобранных окнах считаем dense flow (Farneback), кодируем как 3-канальное изображение → ResNet18. Конкат с RGB-фичами для финальной классификации.

**Зачем.** Flow отлично разделяет hook/uppercut/jab (разная траектория) и `landed` vs `miss` (есть/нет реакции тела соперника).

**Стоимость.** 1 день.

**Ожидаемый эффект.** +0.02-0.04.

### H9 — Pseudo-labels на тренировочных любительских видео ★★★

**Что.** Получив сильную модель на турнирных данных, прогоняем по тренировочным записям (`бокс/IMG_*.mp4`), берём только predictions с confidence > 0.95, добавляем в train. Self-training.

**Зачем.** Расширяет распределение, регуляризует.

**Стоимость.** 1 цикл = +10ч обучения.

**Ожидаемый эффект.** +0.01-0.03.

### H10 — Ансамбль 3-5 моделей с разными inductive bias ★★★★

**Что.**
- Pose-TCN (H1)
- E2E-Spot RGB (H2)
- VideoMAE-classifier на кандидатах (H3)
- Glove-tracker (H4)

Усреднение per-frame score (мягкое) перед peak-picking. Для атрибутов — soft voting с весами по validation.

**Зачем.** Разные ошибки коррелируют слабо, особенно pose ↔ RGB.

**Стоимость.** Только инфер + калибровка.

**Ожидаемый эффект.** +0.03-0.07.

### H11 — Калибровка порогов под FP-штраф ★★★★

**Что.** Grid-search per-video на CV: threshold для peak detection ∈ [0.2, 0.8]. Метрика рассчитывается напрямую. Выбираем порог, максимизирующий final score, **не F1**. Из-за FP-penalty оптимум обычно при precision ~0.75-0.85.

**Зачем.** Дешевле любого нового бэкбона.

**Стоимость.** 1 час.

**Ожидаемый эффект.** +0.02-0.05.

### H12 — Test-time augmentation ★★

**Что.** Горизонтальный flip (но тогда swap red↔blue и left↔right). Multi-scale crops. Усреднение per-frame logits.

**Стоимость.** ×2-3 inference.

**Ожидаемый эффект.** +0.01-0.02.

### H13 — Domain-specific aug ★★★

**Что.**
- Цвет: HSV jitter, **color-swap aug** (поменять local hue red↔blue в обучении 50% батчей → модель не зависит от цвета).
- Crop: random ring-crop с сохранением обоих бойцов.
- Mixup/Cutmix на временной оси.
- Перемешивание видео в батче (не обрезать раунды).

**Зачем.** На 63 видео переобучение неминуемо. Регуляризация даёт +0.02-0.05.

### H14 — Effectiveness через реакцию соперника ★★

**Что.** Отдельная мини-сеть, которая смотрит на opponent (НЕ атакующего) в окне ±10 кадров. landed = резкое движение головы назад, block = поднятие перчаток, miss = нет реакции корпуса.

**Стоимость.** 1 день.

**Ожидаемый эффект.** +0.02-0.04 к `effectiveness`.

### H15 — Уменьшить FP через "no-punch" пуш ★★★★

**Что.** Намайнить hard negatives: кадры замаха, защиты, клинча. Добавить их как "negative" в обучении бинарного детектора. Можно использовать видео-кадры в радиусе 20 кадров от GT, не являющиеся GT.

**Зачем.** FP-penalty — большой штраф. Сейчас сеть будет триггерить на замахи.

**Стоимость.** 1 день.

**Ожидаемый эффект.** +0.03-0.06.

## 5. Постпроцесс

1. **Temporal NMS** на per-frame score. `min_distance ≈ 8-12 кадров` (≈ 0.3 с) — экспериментально.
2. **Soft NMS** (понижение, не отбрасывание) для близких ударов — на брейках бывают двойки/тройки с шагом ~0.3 с.
3. **Fighter consistency:** если за окно 5 кадров детектор выдал два пика с разным fighter, обычно один из них шум. Объединять с argmax score.
4. **Snap to wrist-velocity peak:** взять предсказанный frame, в окне ±5 кадров найти локальный максимум скорости wrist по pose → сдвинуть. Снижает `time_error`.
5. **Fp-management:** sort predictions по score, обрезать снизу так, чтобы соотношение pred/GT_estimated ≈ 0.9 (через калибровку на train).
6. **Атрибут-смузинг по треку:** prior на `hand` от глобальной статистики бойца (right-handed vs left-handed в раунде).
7. **clear=false padding:** в submission всегда выдавать ровно 1594 строки, лишние помечать `clear=false` — бесплатно.

## 6. Тренировки: что где

### Локально (8 ГБ RAM, без GPU)

**Можно:**
- Pose feature extraction (YOLO11n-pose на CPU/GPU слабом).
- Skeleton-TCN/LSTM (H1) — <100 МБ модель, обучается на CPU за часы.
- Glove tracker inference, peak-picking, постпроцесс, валидация.
- Всё связанное с CSV/I/O, ансамбли, поиск порогов.

**Нельзя:**
- VideoMAE/Swin/E2E-Spot full pretrain.
- Дообучение крупных RGB-моделей.

### Облако (что арендовать)

| Задача | GPU | Время | Примерная стоимость |
|---|---|---|---|
| E2E-Spot fine-tune | 1× A100 40GB или RTX 4090 | 12-24 ч | $20-50 |
| VideoMAE-S fine-tune | 1× A100 | 6-12 ч | $15-25 |
| Pose extraction full dataset (YOLOv8-pose-x) | RTX 4090 | 4-8 ч | $5-10 |
| Hyperparam-search (10 runs) | 1× 4090 | 24 ч | $15 |

Платформы: vast.ai, runpod.io, immers.cloud (RU), selectel. **Хранить артефакты в Google Drive / S3** — видео не качать каждый раз.

### Pose-extraction стратегия

Это **узкое место**. ~63+9 = 72 видео × ~3600 кадров × CPU-pose = десятки часов. Решения:
- Кэшировать в `.npz` per video (keypoints + bbox + track_id).
- Использовать YOLOv8-pose-x один раз на GPU в облаке.
- Дополнить **DWPose/RTMW** только для качества кистей.

## 7. Roadmap (6 этапов)

### Этап 0 — Инфра и метрика (1 день)
- Локальная реализация метрики 1:1 → CV на 5 fold по `fight_folder` (важно: split по бою, не по раунду — утечка).
- Проверить fps mapping (H6).
- Перегнать baseline и зафиксировать reference score.

### Этап 1 — Sanity baseline (1-2 дня)
- Pose-LSTM из `boxing_lstm_pipeline/` дообучить на soft Gaussian-targets (H1 lite).
- Калибровка порогов (H11).
- Цель CV: ~0.30-0.40.

### Этап 2 — Сильный pose-pipeline (3-5 дней)
- DWPose/RTMW для всего датасета (один прогон в облаке).
- Glove tracker (H4).
- Полный multi-task TCN (H1).
- Стабильный fighter assignment (H5).
- Цель CV: 0.45-0.55.

### Этап 3 — RGB модель (5-7 дней, облако)
- E2E-Spot или T-DEED на 720p crops (H2).
- VideoMAE classifier на кандидатах (H3).
- Цель CV: 0.55-0.62.

### Этап 4 — Тонкая настройка (3-5 дней)
- Hard-negative mining (H15).
- Flow-stream (H8).
- Effectiveness через реакцию (H14).
- Class-balanced loss (H7).
- Pseudo-labels (H9).

### Этап 5 — Ансамбль + сабмит-инжиниринг (2-3 дня)
- H10 + H11 + H12.
- Калибровка порогов под FP штраф на каждом виде test-боя отдельно.
- Snap to wrist velocity (постпроцесс).

## 8. Чеклист "Что обязательно проверить перед сабмитом"

- [ ] fps мапится в "30 fps frame" (H6).
- [ ] В submission ровно 1594 строки.
- [ ] Лишние строки помечены `clear=false`.
- [ ] `fighter`, `hand`, `target` — строки в нижнем регистре, ровно из словаря.
- [ ] `effectiveness` ∈ {landed, blocked, miss}, не {hit, block}.
- [ ] `id` идёт 1..N подряд.

## 9. Что НЕ стоит делать

- Тренировать с нуля RGB на 63 видео без претрейна — переобучение неминуемо.
- Полагаться на чистый baseline-цвет для `fighter` — тренировочные клипы имеют нейтральную форму.
- Использовать `min_distance=30` в peak-picking — соседние удары близки.
- Сохранять только `clear=true` строки и отдавать <1594 — может ломать формат.
- Считать F1 целью — метрика другая, FP штрафуется агрессивнее.
- Игнорировать venue-shift между training-камерой и tournament-камерой.
